# RAG-002: 샘플 유사도 검증 (P1-5)

**작성일:** 2026-05-28
**검증 방법:** Upstage solar-embedding-1-large 벡터 간 코사인 유사도
**검증 도구:** `EmbeddingPipeline#verifySimilarity(int sampleCount, int topK)`
**상태:** 코드 구현 완료 / 실측 결과는 운영 환경(DB + API 키)에서 실행 후 갱신

---

## 1. 검증 설계

### 1.1 검증 가설

- **H1**: 동일 법령 내 인접 조항은 cosine ≥ 0.6 (도메인 유사성)
- **H2**: 동일 위반 유형(예: 질병 예방·치료 표방) 청크는 법령이 달라도 cosine ≥ 0.5
- **H3**: 무관한 법령 간 청크는 cosine < 0.4 (변별력)

### 1.2 측정 방식

- DB 내 모든 임베딩(127건)을 메모리 로드
- 시드 청크 5건 균등 추출 (전체 길이를 step 으로 나눠 분산)
- 각 시드에 대해 나머지 126건과 cosine 계산 후 top-5 추출
- 자기 자신은 제외

### 1.3 코사인 유사도 공식

```
cos(a, b) = (a · b) / (||a|| × ||b||)
       a · b   = Σ a[i] × b[i]
       ||a||   = sqrt(Σ a[i]²)
```

코드: `EmbeddingPipeline#cosine(float[], float[])` (Brute-force, 4096차원 × 127건 ≈ 50ms 예상)

## 3. 예상 유사도 패턴

### 3.1 정상 작동 시 기대값

| 시나리오 | 예상 top-1 cosine | 예상 top-5 cosine 범위 |
|---|---:|---:|
| 동일 법령 인접 조항 | 0.75 ~ 0.92 | 0.60 ~ 0.92 |
| 동일 위반 유형, 다른 법령 | 0.55 ~ 0.75 | 0.40 ~ 0.75 |
| 다른 도메인 (가이드라인 vs 규정) | 0.45 ~ 0.65 | 0.30 ~ 0.65 |
| HTML 스텁 청크 간 (텍스트가 거의 동일) | 0.95+ | 0.95+ |

### 3.2 이상 신호 (실패 조건)

- 동일 법령 인접 조항 top-1 cosine < 0.5 → 청크 텍스트 노이즈 과다 의심
- 무관한 청크 cosine > 0.7 → 모델 변별력 부족 (Phase 2 BGE-m3 대체 검토)
- top-K 결과가 자기 자신 → 검증 코드 버그 (현재 코드는 `!seed.getId().equals(r.getId())` 필터 적용)

## 4. 실행 절차

### 4.1 사전 조건

1. `embedding_stats.md §4`의 사전 준비 절차 완료 (Docker, API 키, 파이프라인 실행)
2. `GET /api/v1/admin/embedding/pipeline/stats` 응답에서 `embedded == total == 127` 확인

### 4.2 검증 실행

```bash
curl "http://localhost:8080/api/v1/admin/embedding/pipeline/verify-similarity?samples=5&topK=5" | jq > /tmp/verify.json
```

### 4.3 결과 갱신

`/tmp/verify.json` 응답을 받아 본 문서의 §5 표를 채운다. 응답 스키마:

```json
[
  {
    "seedId": 1,
    "seedLaw": "...",
    "seedArticle": "...",
    "seedTextPreview": "...",
    "top": [
      {"id": 12, "lawName": "...", "articleNumber": "...", "preview": "...", "score": 0.8123},
      ...
    ]
  },
  ...
]
```

> 또는 코드 레벨에서 `SampleSimilarity#toMarkdown()` 호출 결과를 그대로 붙여넣을 수 있다.
