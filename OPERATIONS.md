# ⚙️ OPERATIONS — ai-compliance-guard 서비스 운영 가이드

> 서비스를 신뢰성 있고 안전하게 운영하기 위한 운영 정책·프로세스.
> 개발 순서는 [ROADMAP.md](ROADMAP.md), 시장 전략은 [STRATEGY.md](STRATEGY.md) 참고.

컴플라이언스 제품은 "한 번 만들면 끝"이 아니라 **규정·품질·신뢰를 계속 관리**해야 가치가 유지된다. 아래 6개 영역으로 운영 체계를 정의한다.

| 영역 | 핵심 질문 | 도입 시점 |
| :--- | :--- | :--- |
| 1. 규정 데이터 운영 ⚠️ | 규정이 항상 최신인가? | Phase 1 |
| 2. AI 품질 운영 | 판정이 계속 정확한가? | Phase 1~2 |
| 3. LLM 비용 운영 | 검사 1건당 마진이 남는가? | Phase 2 |
| 4. 인프라/서비스 운영 | 서비스가 안정적으로 떠 있는가? | Phase 0~3 |
| 5. 법적 리스크 관리 ⚠️ | 우리가 법적으로 보호받는가? | Phase 2~3 |
| 6. 고객 지원 운영 | 문제를 빠르게 해결하는가? | 런칭 후 |

---

## 1. 규정 데이터 운영 (Regulatory Data Ops) ⚠️ 최우선

**왜 중요한가**: 법령·가이드라인은 계속 개정된다. 오래된 규정으로 판정하면 제품 신뢰가 즉시 붕괴되고, 잘못된 판정은 고객에게 실질적 손해를 입힌다. 컴플라이언스 제품의 생명선.

- **데이터 소스 인벤토리**: 규정별 출처·소관 부처·갱신 주기·최종 반영 시행일을 표로 관리.
- **개정 모니터링**: 국가법령정보센터, 식약처 공지·보도자료, 협회 자율심의 기준 변경을 추적. 초기에는 정기 수동 점검 체크리스트 → 이후 자동 알림화.
- **갱신 파이프라인**: 개정 감지 → 원문 재수집 → 재파싱·재청킹 → 임베딩 재생성 → 평가 세트로 검증 → 배포. 각 규정 청크에 **시행일·버전 메타데이터** 부착.
- **버전 관리·감사 추적**: "어떤 검사 결과가 어떤 규정 버전으로 판정됐는지" 추적 가능해야 한다(분쟁 대비).
- **정기 전수 점검**: 분기별로 전체 규정의 최신성·정확성 검토.

## 2. AI 품질 운영 (AI Quality Ops)

**왜 중요한가**: 모델·프롬프트·규정 데이터가 바뀔 때마다 판정 품질이 흔들릴 수 있다. 품질을 "측정 가능한 지표"로 관리한다.

- **골든 평가 세트**: 위반/정상 광고 카피 + 정답 라벨(위반 유형·근거 조항). 위반 유형별 커버리지 확보.
- **정기 회귀 평가**: 검색 정확도(Recall@K), 에이전트 판정 정확도, 환각률, 거짓양성·거짓음성률을 주기적으로 측정.
- **환각 모니터링**: 비판관(Critique) 에이전트가 걸러낸 케이스를 로깅·분석해 패턴 파악.
- **변경 관리**: 프롬프트·모델·임베딩 변경 시 평가 세트 재실행 후 이전 버전과 비교(회귀 방지).
- **사용자 피드백 루프**: 오판정 신고 → 분석 → 평가 세트 보강 → 다음 회귀 평가에 반영.

## 3. LLM 비용 운영 (Cost Ops)

**왜 중요한가**: LLM API 호출이 최대 변동비. 검사당 원가가 요금보다 높으면 사업이 성립하지 않는다.

- **비용 구조 파악**: 검사 1건 = 임베딩 + 멀티에이전트(검사·비판·교정) + 순환 루프 반복 + 재시도. 토큰 사용량을 분해해 측정.
- **검사당 원가 추적**: 평균 원가를 상시 측정하고 [STRATEGY.md](STRATEGY.md)의 요금제 가격과 마진 비교.
- **최적화 수단**: 프롬프트 캐싱, 컨텍스트 길이 제어, 모델 라우팅(쉬운 케이스는 저가 모델), 순환 루프 반복 상한.
- **비용 가드레일**: 일·월 사용량 임계치 알림, 비정상 호출량 차단.
- **모델 가격 변동 대응**: LLM 공급사 가격·정책 변경 시 원가·요금제 재검토.

## 4. 인프라/서비스 운영 (Service Ops)

- **모니터링**: 가용성, 응답시간, 에러율, 작업 큐 적체, 외부 LLM API 장애 감지.
- **알림/온콜**: 임계치 초과 시 알림 채널, 대응 담당 정의.
- **인시던트 대응**: 심각도 등급 정의, 대응 절차, 사후 리뷰(postmortem).
- **백업/복구**: DB(사용자·이력·규정 데이터) 백업 주기와 복구 테스트.
- **배포**: 환경 분리(local/staging/prod), 무중단 배포, 롤백 전략.
- **로깅 분리**: 운영 로그와 사용자 입력 데이터를 분리 저장(프라이버시 보호, → 5번 항목 연계).

## 5. 법적 리스크 관리 (Legal Risk Ops) ⚠️ 최우선

**왜 중요한가**: 본 서비스의 판정이 "법률 자문"으로 오인되면, 오판정 시 책임 분쟁이 발생할 수 있다. 위반을 놓치거나(거짓음성) 과잉 경고하는(거짓양성) 두 방향 모두 리스크.

- **디스클레이머**: 리포트·UI·약관에 "본 결과는 참고용이며 법률 자문이 아님, 최종 책임은 사용자 및 법률 전문가 검토에 있음"을 명확히 노출.
- **이용약관·면책 조항**: 서비스 한계, 책임 제한, 데이터 처리 범위를 약관에 명시. 런칭 전 변호사 검토 권장.
- **데이터 처리 컴플라이언스**: 개인정보보호법 준수 — 고객 입력 데이터를 외부 LLM에 전송하는 것에 대한 동의·고지, 보관·파기 정책.
- **도입 시점**: 디스클레이머는 Phase 2 UI부터 노출, 정식 이용약관은 Phase 3 런칭 전 필수.

## 6. 고객 지원 운영 (Support Ops) — 런칭 후

- **문의 채널·SLA**: 문의 접수 경로와 응답 목표 시간.
- **FAQ**: 반복 문의를 문서화.
- **오판정 신고 처리**: 신고 접수 → 검토 → 고객 회신 → 2번(AI 품질 운영) 평가 세트로 환류하는 흐름.

---

## 운영 성숙도 진행

운영 항목은 한 번에 다 갖추지 않고 개발 단계에 맞춰 점진적으로 도입한다.

- **Phase 0~1**: 규정 데이터 운영 프로세스 + 골든 평가 세트 기반 마련, 기본 인프라 모니터링.
- **Phase 2**: AI 품질 회귀 평가 정례화, LLM 비용 추적 시작, 디스클레이머 노출.
- **Phase 3**: 법적 약관 확정, 인시던트 대응 체계, 비용 가드레일, 고객 지원 채널 가동.
