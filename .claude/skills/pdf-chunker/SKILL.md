## 핵심 역할

P1-2 작업을 담당한다. `_workspace/REGULATIONS_INVENTORY.md`에서 확인된 6개 PDF 규정을 읽고, 조항 단위로 파싱하여 300~500 토큰 크기의 청크로 분할한다. 각 청크에는 법령명, 조항번호, 시행일, 출처 파일명 등의 메타데이터를 부착하고, JSONL 형식으로 저장하여 임베딩 파이프라인의 입력이 되도록 준비한다.

**목표**: `_workspace/regulations_chunks.jsonl` 파일 생성. 행당 하나의 JSON 객체로, 청크 텍스트 + 메타데이터를 포함. 샘플 20건을 수동 검증하여 청크 품질 확인.

## 작업 원칙

- **조항 구조 보존**: 한국 법령은 조(條), 항(項), 호(號)의 계층 구조를 가진다. 파싱 시 이 구조를 파괴하지 않고, 완전한 문장 또는 논리 단위로 청킹한다.
- **메타데이터 정확성**: 각 청크는 어느 법령의 몇 조 몇 항인지 명확히 기록. 향후 근거 조항 검증에 쓰인다.
- **토큰 효율성**: OpenAI text-embedding-3 기준 약 300~500 토큰. 너무 짧으면 정보량 부족, 너무 길면 임베딩 비용 상승 + 시맨틱 퍼짐.
- **재현 가능성**: 청킹 알고리즘(정규식 규칙, 분할 포인트)을 문서화하여 새 규정 추가 시 동일 기준으로 적용 가능하게 한다.

## P1-2 작업 상세

### 1단계: PDF 텍스트 추출

도구 선택 기준:
- **Apache Tika**: 복잡한 PDF 레이아웃 처리, 이미지 OCR 미지원
- **PDFBox**: Java 기반, pgvector 프로젝트와 같은 환경에서 통합 가능
- **pdfplumber**: Python 기반, 테이블 인식 강함, 간단한 설정

**선택 기준**: 규정 문서가 대부분 텍스트 기반이고 테이블이 적으므로 **pdfplumber** 사용 권장. 더 간단하고, 정규식으로 조항 구조를 쉽게 인식할 수 있다.

### 2단계: 조(條)·항(項)·호(號) 구조 파싱

한국 법령의 표준 구조:

```
제N조 (조 제목)
① (1항) 내용...
② (2항) 내용...
    1. (1호) 세부 내용...
    2. (2호) 세부 내용...
③ (3항) 내용...
```

정규식 규칙:
- 조: `^제\d+조(?:\s|$)` (줄 시작에서 "제N조")
- 항: `^[①②③④⑤⑥⑦⑧⑨⑩]|^\s*\d+\)\s*` (원문자 또는 숫자+괄호)
- 호: `^\s+[가-힣]\.\s|^\s+\d+\.\s` (들여쓰기 + 한글문자/숫자 + 점)

### 3단계: 청킹 전략

**기본 전략**: 조 단위로 시작. 조 하나가 500토큰을 초과하면 항 단위로 분할. 항도 500토큰 초과 시 호 단위로 분할.

**예시**:
```
조 1: 200토큰 → 그대로 청크 1 (법령명 + 제1조 전체)
조 2: 800토큰 → 항 단위로 분할
  - 청크 2: 제2조 ① (250토큰)
  - 청크 3: 제2조 ② (300토큰)
  - 청크 4: 제2조 ③ (250토큰)
```

**토큰 계산**: Python tiktoken 라이브러리 사용 (OpenAI cl100k_base 인코더).

### 4단계: JSONL 파일 생성

각 행당 하나의 청크:
```json
{
  "id": "건강기능식품법_제12조_①_001",
  "law_name": "건강기능식품에 관한 법률",
  "article": "제12조",
  "section": "①",
  "subsection": null,
  "text": "건강기능식품은 질병의 예방·치료를 목적으로 표시·광고할 수 없다.",
  "source_file": "regulations/건강기능식품법_20260526.pdf",
  "enactment_date": "2005-01-01",
  "last_revised": "2024-12-31",
  "tokens": 45,
  "order": 1
}
```

**필드 설명**:
- `id`: 고유 식별자 (법령명_조항_항_순서)
- `law_name`: 법령의 공식 명칭
- `article`, `section`, `subsection`: 조·항·호 번호
- `text`: 청크 텍스트 (원문 그대로, 과도한 정제 금지)
- `source_file`: 원본 PDF 파일명
- `enactment_date`: 법령 공식 시행일
- `last_revised`: 최신 개정일
- `tokens`: 이 청크의 토큰 수 (검증용)
- `order`: 해당 조 내에서의 순서

### 5단계: 샘플 검증

전체 청크 중 무작위로 20건을 샘플링하여 수동 검증:
- [ ] 청크가 완전한 문장으로 끝나는가?
- [ ] 메타데이터가 정확한가?
- [ ] 토큰 수가 300~500 범위인가? (범위 벗어난 항목 체크)
- [ ] 법령의 논리적 의미가 손상되지 않았는가?

## 입력/출력 프로토콜

**입력:**
- Phase 1 Orchestrator의 DATA-002 작업 할당
- `_workspace/REGULATIONS_INVENTORY.md` (규정 인벤토리, 파일 경로 확인용)

**출력:**
- `_workspace/regulations_chunks.jsonl` (청크 데이터셋, 전체 행 수 기록)
- `_workspace/CHUNKING_REPORT.md` (청킹 통계: 총 조 수, 총 청크 수, 평균 토큰, 토큰 분포)
- `_workspace/sample_chunks_validation.md` (샘플 20건 검증 결과)

**검증:**
- 청크 파일이 JSONL 형식 (각 행이 유효한 JSON)
- 모든 청크가 300~500 토큰 범위 (범위 외 청크 목록 기록)
- 메타데이터 완결성 (null 허용 안 함, `id` 중복 없음)
- 샘플 20건 수동 검증 완료

## 에러 핸들링

| 상황 | 대응 |
|------|------|
| PDF 텍스트 추출 실패 | 파일 손상 여부 확인, OCR 필요시 보고, 다른 형식 대체 가능성 검토 |
| 조항 파싱 실패 (정규식 미매칭) | 문서의 실제 형식 확인, 정규식 규칙 갱신, 수동 검토 필요 구간 기록 |
| 청크가 너무 긴 경우 (1000+ 토큰) | 호 단위로 재분할, 최대 길이 제한 구현 |
| 청크가 너무 짧은 경우 (<100 토큰) | 이전 청크와 병합 검토 (논리 구조 손상 확인 후) |
| 메타데이터 정보 누락 | 인벤토리에서 재확인, 확인 불가 시 "미확인" 표시 |

## 테스트 시나리오

### Happy Path
```
1. 6개 PDF 모두 텍스트 추출 성공
2. 모든 조항이 정규식으로 정확히 파싱됨
3. 청크가 300~500 토큰 범위에 분포
4. regulations_chunks.jsonl 생성 (예: 2,000~3,000행)
5. 샘플 20건 검증 완료, 품질 확인
```

### 엣지 케이스
```
1. 스캔 PDF (OCR 필요): 이미지 감지 후 Data Engineer에 보고
2. 특수 문자·도표: 원문 보존, 도표는 텍스트 설명으로 변환
3. 조항 번호 특수 형식: 정규식 규칙 갱신, 재파싱
4. 빈 조항: 텍스트 없는 조항 제외 또는 "내용 없음" 마크
```
