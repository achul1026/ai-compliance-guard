## 🔑 선행 기술 결정 사항 (Phase 0~1 진입 전 확정)

아래 항목은 구현 도중 번복하면 비용이 큰 결정들이다. 해당 단계 진입 시 함께 논의해 확정한다.

- **D1. Gradle 모듈 구조**: 초기에는 단순하게 시작하고 필요 시 분할 (Phase 0에서 확정).
- **D2. 한국어 BM25 구현 방식**: PostgreSQL 기본 full-text search는 진정한 BM25가 아니고 한국어 형태소 분석이 약함. 후보 — (a) `pg_search`(ParadeDB) 확장, (b) 별도 검색엔진(OpenSearch/Elasticsearch + nori), (c) 애플리케이션 레벨 BM25(Lucene). Phase 1에서 확정.
- **D3. 임베딩 모델**: 한국어 법령/규정 특화 필요. 후보 — OpenAI text-embedding-3, Upstage solar-embedding, BGE-m3 등. Phase 1에서 확정.
- **D4. Re-ranking 방식**: Cohere Rerank, Upstage, cross-encoder 등. Phase 1에서 확정.
- **D5. LLM 엔진**: OpenAI GPT-4o vs Upstage Solar (한국어 법률 추론 품질 비교 후 결정). Phase 2에서 확정.
- **D6. 에이전트 오케스트레이션**: LangChain4j vs LangGraph (Java 생태계 호환성 검토). Phase 2에서 확정.

## 📅 Phase 0 — 프로젝트 기반 셋업

> 목표: 코드를 작성할 수 있는 빌드/실행 환경을 갖춘다.

- [ ] **P0-1. Gradle 멀티모듈 골격 생성**
  초기 모듈안: `:api`(Spring Boot 웹/REST), `:agent`(에이전트 파이프라인), `:rag`(검색), `:infra`(영속성·외부 API), `:common`(공용 DTO/유틸).
  → 검증: `./gradlew build` 성공.
- [ ] **P0-2. Spring Boot 애플리케이션 부트스트랩**
  `:api` 모듈에 `@SpringBootApplication`, 헬스체크 엔드포인트(`GET /health`) 추가.
  → 검증: 앱 기동 후 `/health`가 200 응답.
- [ ] **P0-3. 로컬 인프라 구성 (Docker Compose)**
  PostgreSQL + pgvector 확장 컨테이너 정의.
  → 검증: `docker compose up` 후 DB 접속 + `CREATE EXTENSION vector` 성공.
- [ ] **P0-4. 설정·시크릿 관리 체계**
  `application.yml` 프로파일 분리(local/prod), LLM API 키 등 환경변수 주입 구조. `.gitignore` 정비.
  → 검증: 로컬 프로파일로 앱이 DB에 연결됨.

## 📅 Phase 1 — 데이터 구축 + Hybrid RAG (1~2개월)

> 목표: "이 광고 문구가 위반되는가?"라는 질의에 정확한 근거 조항을 찾아오는 검색 API를 완성한다.

### 1-A. 규정 데이터 수집·정제
- [ ] **P1-1. 근거 규정 소스 확정 및 수집**
  대상: 식품 등의 표시·광고에 관한 법률(시행령/시행규칙 포함), 건강기능식품에 관한 법률, 화장품법, 식약처 표시·광고 가이드라인, 부당 표시·광고 FAQ, 협회 자율심의 기준.
  → 검증: 원문 PDF/문서 목록과 출처가 정리됨.
- [ ] **P1-2. PDF 파싱 및 청킹 전략 수립**
  조(條)·항(項)·호(號) 단위 구조를 보존하는 청킹. 메타데이터(법령명, 조항번호, 시행일) 부착.
  → 검증: 샘플 법령이 조항 단위로 올바르게 분할되고 메타데이터가 채워짐.
- [ ] **P1-3. 위반 유형 택소노미 정의**
  예: 질병 예방·치료 효능 표방, 의약품 오인, 거짓·과장, 객관적 근거 없는 표현, 소비자 기만 등 분류 체계.
  → 검증: 위반 유형별로 대표 예시·근거 조항이 매핑된 표 완성.

### 1-B. Hybrid RAG 파이프라인
- [ ] **P1-4. DB 스키마 설계**
  규정 청크 테이블(본문, 메타데이터, 임베딩 vector 컬럼), 인덱스 설계.
  → 검증: 마이그레이션 적용 + 샘플 데이터 적재 성공.
- [ ] **P1-5. 임베딩 파이프라인 (D3 확정)**
  청크 → 임베딩 생성 → pgvector 적재 배치 작업.
  → 검증: 전체 청크가 임베딩되어 벡터 검색 쿼리가 동작.
- [ ] **P1-6. 키워드 검색(BM25) 구현 (D2 확정)**
  조항 번호·고유명사 매칭용 키워드 검색.
  → 검증: "식품표시광고법 제8조" 같은 질의가 정확히 해당 조항을 반환.
- [ ] **P1-7. 하이브리드 결합 + Re-ranking (D4 확정)**
  벡터 + 키워드 결과 병합 후 관련도 재정렬.
  → 검증: 테스트 질의 세트에서 정답 조항이 상위 N개 안에 포함.
- [ ] **P1-8. 검색 API 완성**
  `POST /api/search` — 질의 입력 → 관련 조항 컨텍스트 반환.
  → 검증: API 통합 테스트 통과, 검색 정확도 평가 세트로 측정.

## 📅 Phase 2 — Multi-Agent 파이프라인 + UI (1개월)

> 목표: 광고 문구를 입력하면 위험 구절 하이라이팅 + 대안 제시 리포트를 출력한다.

### 2-A. 에이전트 오케스트레이션
- [ ] **P2-1. 오케스트레이션 프레임워크 도입 (D5·D6 확정)**
  → 검증: 단순 단일 에이전트 호출이 엔드투엔드로 동작.
- [ ] **P2-2. 검사관(Auditor) 에이전트**
  검색 컨텍스트 기반으로 위반 소지·리스크 리포트 생성.
  → 검증: 위반 샘플 입력 시 위반 구절·근거 조항·위험도가 출력.
- [ ] **P2-3. 비판관(Critique) 에이전트**
  Auditor가 인용한 근거 법령의 실재성·정합성 교차 검증(환각 차단).
  → 검증: 의도적으로 잘못된 근거를 심으면 Critique가 탐지.
- [ ] **P2-4. 교정관(Remediator) 에이전트**
  안전한 대체 문구 제안.
  → 검증: 위반 구절마다 규정에 맞는 대안 문구가 생성.
- [ ] **P2-5. Stateful 워크플로우 + 순환 루프**
  Critique가 오류 감지 시 Auditor로 피드백하는 루프 제어, 무한루프 방지 한계 설정.
  → 검증: 피드백 루프가 동작하고 최대 반복 횟수에서 안전하게 종료.
- [ ] **P2-6. 분석 API 완성**
  `POST /api/audit` — 광고 문구 입력 → 최종 컴플라이언스 리포트.
  → 검증: 엔드투엔드 통합 테스트 통과.

### 2-B. 프론트엔드
- [ ] **P2-7. 입력 UI**
  텍스트 입력 / 문서 업로드 화면.
  → 검증: 입력이 분석 API로 전송됨.
- [ ] **P2-8. 결과 대시보드**
  위험 구절 하이라이팅(위험도별 색상 차등), 대안 툴팁 표시.
  → 검증: 리포트가 시각적으로 올바르게 렌더링됨.

## 📅 Phase 3 — 엔터프라이즈 보안 + 수익화 (1개월)

> 목표: B2B 고객이 안심하고 결제해 쓸 수 있는 서비스로 런칭한다.

- [ ] **P3-1. 인증/세션 보안**
  JWT(HttpOnly Cookie) 기반 인증, 사용자·분석 이력 관리.
  → 검증: 인증 플로우 통합 테스트 통과.
- [ ] **P3-2. 데이터 프라이버시 강화**
  외부 LLM 호출 시 `data_collection: false` 옵션 처리, 입력 데이터 마스킹, 분석 이력 AES-256 암호화 저장.
  → 검증: LLM 요청 페이로드에 학습 거부 옵션 적용 확인, 저장 데이터 암호화 확인.
- [ ] **P3-3. 사용량 제한 (Free Tier)**
  월 3~5회 무료 검사 카운팅·제한.
  → 검증: 한도 초과 시 차단되고 안내 메시지 노출.
- [ ] **P3-4. 결제 연동 (Paid Tier)**
  토스페이먼츠 연동 — 건당 크레딧 차감 또는 월간 구독.
  → 검증: 테스트 결제 성공 + 크레딧/구독 상태 반영.
- [ ] **P3-5. 런칭 준비**
  랜딩 페이지("절대 유출되지 않습니다" 보안 마케팅 전면 배치), 배포 파이프라인, 모니터링.
  → 검증: 운영 환경 배포 후 핵심 플로우 동작 확인.
